**Critical Zod v4 notes:**
- Use `z.string().min(1)` NOT `z.string().nonempty()` (deprecated in v4)
- Use `z.coerce.number()` for PORT env var (env vars are always strings)
- `z.enum()` for NODE_ENV

```typescript
import 'dotenv/config';
import { z } from 'zod';

const EnvSchema = z.object({
  DATABASE_URL: z.string().min(1, 'DATABASE_URL is required — set it in apps/server/.env'),
  PORT: z.coerce.number().default(3001),
  CLIENT_URL: z.string().default('http://localhost:3000'),
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
});

// Throws with descriptive Zod error at startup if any required env var is missing or invalid.
// Do NOT wrap in try/catch here — fail fast is the intended behavior.
export const config = EnvSchema.parse(process.env);

export type Config = z.infer<typeof EnvSchema>;
```

**Where does .env live?** `dotenv/config` reads `process.cwd()/.env`. When `tsx watch src/index.ts` runs from `apps/server/`, it reads `apps/server/.env`. Document this in a comment above the import.

**Fastify v5 notes:**
- Use `await server.register(cors, {...})` — v5 uses Promise-based registration
- `server.listen()` takes an options object `{port, host}` — NOT separate positional args
- Set `host: '0.0.0.0'` so the server is reachable from the browser
- Logger is Pino (bundled with Fastify) — `logger: true` enables structured JSON logging

**GET /health schema note:** Include a JSON schema for the 200 response. Fastify uses this for serialization optimization and documentation. The response must be exactly `{"status": "ok"}` — no extra fields.

```typescript
import Fastify from 'fastify';
import cors from '@fastify/cors';
import cookie from '@fastify/cookie';
import { config } from './config.js';

const server = Fastify({
  logger: true,
});

// CORS: allow requests from Vite dev server on port 3000
await server.register(cors, {
  origin: config.CLIENT_URL,
  credentials: true,
});

// Cookie plugin: needed by Better Auth (added in v2)
await server.register(cookie);

// Health check — returns exactly {"status":"ok"}
// DO NOT add env info, version numbers, or uptime — information disclosure risk
server.get(
  '/health',
  {
    schema: {
      response: {
        200: {
          type: 'object',
          properties: {
            status: { type: 'string' },
          },
          required: ['status'],
        },
      },
    },
  },
  async () => {
    return { status: 'ok' };
  }
);

// Start server
try {
  await server.listen({ port: config.PORT, host: '0.0.0.0' });
  server.log.info(`Server running on http://0.0.0.0:${config.PORT}`);
} catch (err) {
  server.log.error(err);
  process.exit(1);
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| process.env → config | Untrusted env vars cross the Zod validation boundary |
| client (port 3000) → server (port 3001) | CORS boundary — only trusted origin allowed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | GET /health | mitigate | Response schema locked to `{status: string}` only — no env info, uptime, or version exposed |
| T-02-02 | Spoofing | CORS origin | mitigate | `origin: config.CLIENT_URL` (explicit localhost:3000) — not `origin: '*'`; credentials: true enforces same-origin cookie rules |
| T-02-03 | Tampering | Env validation | mitigate | `EnvSchema.parse(process.env)` throws at startup on missing/invalid vars — no silent fallback to insecure defaults |
| T-02-04 | Denial of Service | Server crash on missing env | accept | Intentional fail-fast behavior; better than silent misconfiguration |
</threat_model>
